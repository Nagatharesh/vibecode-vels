HOW IT WORKS

Request Lifecycle

Every request flows through 4 stages before completing.

01

Request Arrives

A request enters the system with a type (light/normal/heavy/admin) and optional session ID attached.

02

Algorithm Selects Server

The active algorithm evaluates all healthy, non-sleeping servers and picks the optimal target based on CPU, connections, or session affinity.

03

Request Processed

The target server's CPU rises proportional to request cost. Latency is computed. The cool-down mechanism monitors for overload threshold breaches.

04

Server Recovers

After processing, CPU decays back to baseline. Sleeping servers consolidate idle load. The dashboard reflects all state changes in real-time.

ALGORITHMS

4 Routing Strategies

Switch between them live in the simulator to compare performance.

Round Robin 60 / 100

Distributes requests equally in a fixed cycle. Simple and predictable, but ignores actual server load. Best when all servers are identical and traffic is uniform.

STATELESS O(1) NO AWARENESS
Least Load 85 / 100

Routes every request to the server with the lowest current CPU. Adapts dynamically to real load conditions. Best general-purpose algorithm for heterogeneous traffic.

ADAPTIVE CPU-AWARE BEST BALANCE
Weighted 75 / 100

Assigns traffic proportional to each server's capacity weight. Higher-weight servers receive more requests. Ideal when servers have different hardware specs or capacities.

CAPACITY-AWARE CONFIGURABLE PROBABILISTIC
Session-Aware 92 / 100

Maintains sticky session affinity while falling back to least-connections if the pinned server degrades. The best of both worlds — consistency + adaptivity.
